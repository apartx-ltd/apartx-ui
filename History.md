# История изменений — apartx-ui

## 2026-06-21

### Версия 0.1.18

### Добавлено

* **`PageTransition`: проп `slideStyle` (`'auto' | 'ios' | 'shared-axis'`).** Осознанный
  выбор стиля анимации смены страниц вместо неявного дефолта. `'ios'` — горизонтальный
  push-слайд в стиле iOS (входящий слой выезжает поверх, уходящий слегка уезжает под
  него); `'shared-axis'` — Material shared-axis (слой целиком уезжает по оси с
  затуханием). `'auto'` (дефолт) резолвится по ОС через `detectMobileOS`: iOS → `'ios'`,
  остальное → `'shared-axis'`. Стиль прокидывается в DOM как `data-slide-style`, а
  keyframes заскоуплены по этому атрибуту — оба стиля работают рядом без протечек.
* **`detectMobileOS` (`apartx-ui/utils`)** — лёгкий детект мобильной ОС (iOS/Android/other)
  по UA, для резолва `slideStyle='auto'`. Покрыт первым Vitest-харнессом кита.
* **Экспорт типа `SlideStyle`** из `apartx-ui/navigation` для типобезопасного
  использования пропа потребителями.

### Исправлено

* **`shared-axis`: bleed/flash-free.** Затухание применяется ко всему слою (а не только
  к контенту) и на forward-in, и на back-exit — нет вспышки фона и «протечки» соседнего
  слоя на стыке транзишена.

## 2026-06-16

### Версия 0.1.17

### Изменено

* **`Image`: сессионный кэш загруженных картинок (`<script module>`).** Модульный
  `Set<string>` хранит `src`, которые хотя бы раз успешно загрузились. При повторном
  монтировании `<Image>` с уже известным `src` стартует сразу в статусе `loaded`
  (без skeleton, `opacity-100` с первого кадра) вместо повторного прохода
  `loading → loaded`. Раньше каждый ремоунт переигрывал skeleton + 300ms fade, хотя
  байты уже были в кэше браузера — это давало моргание (в списках, при навигации
  туда-обратно). Кэшируются только успешные загрузки, так что ошибки по-прежнему
  идут через повтор. SSR/гидрация не ломаются: на первом клиентском рендере кэш пуст
  (заполняется только после `onload`), выигрыш — на последующих ремоунтах.

## 2026-06-16

### Версия 0.1.16

### Исправлено

* **`RangeCalendar`: локализация названий месяцев и дней недели.** Добавлен опциональный
  проп `locale` (BCP-47, дефолт `'en-US'`), который пробрасывается в `bits-ui`
  `RangeCalendar.Root`. Раньше проп не передавался, и bits-ui (`@internationalized/date`)
  всегда форматировал заголовок месяца и шапку дней недели по дефолтной `'en-US'` —
  календарь оставался английским при любом языке приложения. Дефолт сохраняет прежнее
  поведение, так что для `apartx-admin`/`apartx-cabinet` изменение не ломающее.

## 2026-06-16

### Версия 0.1.15

### Добавлено

* **`Image` — новый компонент (`apartx-ui/display`).** Картинка со skeleton-плейсхолдером
  на время загрузки и состоянием ошибки с кнопкой повтора. Защита от layout-shift через
  bits-ui `AspectRatio` (padding-% хак, безопасно для legacy-браузеров) при заданном `ratio`,
  иначе размер берётся из класса. Пропсы по конвенциям кита: `src`, `alt`, `ratio`, `fit`,
  `loading`, `errorText`, `retryText`, `showError`, `onStatusChange(status)`, `onRetry`,
  `class`, `imgClass`. Кнопка повтора `stopPropagation`'ит, чтобы тап не «проваливался» в
  навигацию родительской карточки.

### Исправлено

* **`Carousel`: пропсы `navigation` / `pagination` / `loop` / `autoplay` передаются булевыми,
  а не строками `'true'`/`'false'`.** `swiper-container` объявляет эти параметры как JS-свойства,
  чей сеттер кладёт значение как есть (без `formatValue`), а Svelte 5 для односложных пропов
  пишет именно свойство. Строка `'false'` — truthy → Swiper рисовал стрелки `<>` навигации и
  включал loop/autoplay, даже когда они выключены. Булевы значения сохраняются как настоящие
  `false`/`true`. (`css-mode` не затронут: его дефисное имя идёт через атрибут, где `'true'`/
  `'false'` парсятся корректно.)
* **`BottomSheet`: горизонтальный свайп вложенного контента (карусель) работает и в нижней
  позиции шторки.** Добавлен axis-lock: на первых ~6px жеста он фиксируется как `'x'`
  (горизонтальный — отдаётся дочернему скроллеру, шторку не тащит и не `preventDefault`'ит)
  или `'y'` (вертикальный — обычный перехват шторки/списка). Раньше ниже верхнего снапа
  `shouldDrag()` перехватывал ЛЮБОЙ тач как драг шторки и `preventDefault`'ил нативный
  горизонтальный скролл карусели (в верхнем снапе работало, т.к. там ветка по направлению
  возвращала false). Вертикальное поведение драга/списка не изменилось.

## 2026-06-15

### Версия 0.1.14

### Добавлено

* **`PageTransition` / `PageLayer`: проп `providePortalHost` (default `true`).** Вложенный
  `<PageTransition>` (контент таба внутри shell) можно пометить `providePortalHost={false}`,
  чтобы `<BottomSheet portalTarget="page">` портировался во ВНЕШНИЙ, полноэкранный слой,
  который реально слайдится при push'е страницы, а не во внутренний (ограниченный футером)
  слой — иначе шторка не «едет» вместе с переходом. admin/cabinet не затронуты (дефолт
  `true` = прежнее поведение).
* **`BottomSheet`: проп `animate` (default `true`).** При `false` present/dismiss мгновенные
  (без слайда и фейда бэкдропа) — чтобы восстановить уже открытую шторку (возврат на карту)
  без повторной анимации; затем хост снова включает анимацию.
* **Google Maps: кластеризация.** Провайдер google лениво подгружает
  `@googlemaps/markerclusterer` (UMD c jsDelivr, как и yandex-кластерер). `ClusterPoint`
  получил `renderKey` — токен внешнего вида: при его смене для существующего `id` маркер
  пере-стилизуется на месте (без смены `id` → набор точек стабилен, слой не
  пере-кластеризуется и не моргает).

### Изменено

* **Mobile-переход — НЕПРОЗРАЧНЫЙ слайд (iOS-style).** Ведущая страница едет на всю ширину
  (`translateX(100%)` → `translateZ(0)`) и полностью кроет уходящую. Settled-transform =
  `translateZ(0)`, а НЕ `none`: слой с `transform:none` перестаёт быть containing block для
  `position:fixed`-потомков, и портированная шторка на один кадр ре-рутится между
  композитными слоями — сквозь неё мелькает фон (напр. нижняя навигация). Раньше был
  30%-слайд + opacity-fade, через полупрозрачность которого просвечивала уходящая страница.
* **`MapClusterer` (fallback-маркеры): реконсиляция по `id`.** Маркеры хранятся в `Map` по
  `id`; при апдейте создаются/удаляются только изменившиеся, а не пересоздаётся весь набор —
  нет мерцания всех пинов при ре-derive `points` (переключение выделения, новые результаты).

## 2026-06-11

### Версия 0.1.13

### Изменено

* **`PageTransition.direction` принимает `() => Direction`** (обратносовместимо — строка
  работает ровно как раньше, admin/cabinet не затронуты). `direction` теперь резолвится
  **лениво** (`resolveDir()`) на enter и outro. Зачем: когда хост, владеющий направлением,
  сносится не своим `{#key}`, а **ANCESTOR-блоком** (вложенный роутер-shell уходит на
  детальный роут), value-prop замораживается на последнем рендере хоста, и уходящий слой
  читает **устаревшее** направление (Svelte `derived_inert`). Функция читает живое
  состояние на outro. `enterKind` получил `key` в зависимости, чтобы функция
  пере-резолвилась на каждую навигацию (ссылка на функцию стабильна). См. `SlotLayout` в
  apartx-spaces.

## 2026-06-10

### Версия 0.1.12

### Добавлено

* **`BottomSheet portalTarget="page"` — оверлей едет вместе со страницей при переходе.**
  По умолчанию (`'body'`) шит порталится в `<body>` (как раньше — admin/cabinet не
  затронуты). При `'page'` порталится в элемент текущего слоя `PageTransition` (его
  отдаёт `PageLayer` через новый контекст `set/getPagePortalHost` в `navigation/context`),
  поэтому при навигации шит уезжает/приезжает СО страницей, а не висит на месте. Нет
  `PageTransition`-предка → fallback на `<body>`.

### Изменено

* **`PageLayer` приведён к контракту «скроллит только `<Content>`».** Слой `.pt-layer`
  стал чистой сценой трансформа (`overflow: visible`, НЕ скролл-контейнер); скролл
  перенесён на внутренний `.pt-content` (`height:100%; overflow-y:auto`), туда же —
  `use:scrollRestore`. Root `PageTransition` сменил `overflow-hidden` → `overflow-clip`
  (клипует слайд, но НЕ скроллируемый). Это устраняет первопричину рывка портального
  `position:fixed`-оверлея: фокус Dialog'а при открытии делал scroll-into-view и дёргал
  скроллируемый трансформ-предок; теперь в цепочке предков скроллера нет вовсе. Страницы
  со своим `<Content>` (admin/cabinet) не меняют поведение — `<Content>` как и прежде
  владеет скроллом, внешний `.pt-content` дормантен.

### Версия 0.1.11

### Добавлено

* **CSS-barrel `apartx-ui/styles`** — один future-proof импорт вместо пофайловых.

  импортят `tokens.css`/`typescale.css` напрямую — мигрируют на barrel по мере удобства).

### Добавлено

  прячущая нативный скроллбар (`scrollbar-width:none` + `-ms-overflow-style` +

## 2026-06-09

### Версия 0.1.7

Новый `BottomSheet` — bottom-sheet со снап-поинтами на bits-ui `Dialog`.

### Добавлено

* **`BottomSheet` — draggable bottom-sheet на bits-ui `Dialog`.** Оболочка — `Dialog`
  (`Portal` в `<body>`, бэкдроп, focus-trap, escape, scroll-lock), консистентно с
  `Dialog`/`AlertDialog`. Слой жеста (snap-поинты как доли `0..1`/px, drag, физика
  релиза по velocity+ближайшему снапу, handoff «тянем шторку ↔ скроллим контент» по
  позиции скролла) — свой, на рунах, адаптирован из vaul-svelte (MIT, huntabyte).
  `Portal` в `<body>` снимает корневую боль cupertino-pane: пейн больше не привязан к
  родителю/хедеру и не зависит от внутренних замеров библиотеки. API: `open` (bindable),
  `snapPoints`, `activeSnapPoint` (bindable), `dismissible`, `backdrop`/`backdropOpacity`,
  `squareCornersAtTop`, `showHandle`, `modal`, `onSnapChange`/`onOpenChange` +
  lifecycle `onWillDismiss` (slide-out start, content still shown) / `onDidDismiss`
  (fully closed & unmounted; cancelled if reopened first) — те же имена, что у
  `CupertinoPane`. Появление/уход анимированы (выезд снизу + затухание бэкдропа).
  `CupertinoPane` пока остаётся; миграция search-шторки — первый потребитель.

### Версия 0.1.6

`CupertinoPane`: не шлёт dismiss-события при размонтировании; не презентует дважды на маунте. `MapView`: хром карты изолирован под оверлеями.

### Исправлено

* **`MapView` — изолирующий stacking context (`isolate`).** Хром провайдера (у Yandex —
  копирайт / ссылка «Открыть Яндекс Карты») пинится с очень высоким z-index и без
  изоляции конкурировал в корневом stacking-контексте → рендерился ПОВЕРХ оверлеев
  приложения (например, шторки `CupertinoPane` на z-50). Корневой `<div>` карты получил
  `isolation: isolate`, поэтому весь хром карты теперь живёт под любым внешним оверлеем
  независимо от внутренних z-index провайдера. Kit-овые M3-контролы (z-10) продолжают
  работать внутри этого контекста.

* **`CupertinoPane` — двойной present на маунте убран.** `createPane()` презентовал
  пейн, и сразу следом show-`$effect` презентовал его ВТОРОЙ раз. Второй вызов
  cupertino `present()` на уже отрендеренном пейне делает `moveToBreak(initialBreak)` —
  то есть затирал любой программный брейк, выставленный хостом из `onDidPresent`
  (например, restore последнего брейка при «назад»). Введён флаг `lastShow` (последнее
  применённое значение show): show-`$effect` презентует/прячет только при РЕАЛЬНОЙ смене,
  а маунтовый прогон, где createPane уже презентовал, пропускается. Для консьюмеров без
  программного moveToBreak поведение не меняется (тот же конечный брейк).

* **`CupertinoPane` — onWillDismiss/onDidDismiss больше не дёргаются при unmount.**
  cupertino `destroy()` эмитит оба dismiss-события (как при свайпе/тапе по бэкдропу), а
  kit зовёт `pane.destroy()` в onMount-cleanup — поэтому при размонтировании компонента
  (например, SPA-переход на другую страницу) хост получал ложный «dismiss». Если хост на
  onDidDismiss чистит сохранённое состояние шторки (для restore при «назад»), оно стиралось
  ровно в момент ухода со страницы. Введён флаг `destroyingOnUnmount` (ставится в
  onMount-cleanup ДО `destroy()`): обёртки onWillDismiss/onDidDismiss при teardown'е
  ни хоста не уведомляют, ни `createPane()` не зовут (re-arm воскрешал бы уходящую шторку).

## 2026-06-08

### Версия 0.1.5

Bottom-sheet поверх карты: тёма/z-index/острые углы `CupertinoPane`, хуки камеры карты и брейка шторки.

### Добавлено

* **`MapView` — `onCameraChange({ center, zoom })`.** Дёргается в конце жеста пана/зума (Yandex: `YMapListener.onActionEnd`, Google: `idle`). Для персиста позиции карты (например, в URL). `setCenter`-эффект `MapView` получил epsilon-guard, чтобы запись камеры обратно в пропы не пере-применяла ту же позицию и не зацикливалась.
* **`CupertinoPane` — `onBreakChange(break)`.** Отдаёт брейк (`top`/`middle`/`bottom`), вычисленный по реальной позиции (`getPanelTransformY()` против `breakpoints.breaks`), а не через библиотечный `currentBreak()`, который рассинхронизируется. Дёргается на drag/transition-settle.
* **`CupertinoPane` — `squareCornersAtTop`.** На верхнем брейке верхние углы шторки становятся острыми (0), на остальных — скруглённые. Завязано на позицию (`getPanelTransformY`), не на ненадёжный `currentBreak()`; обновляется на `onDrag`/`onDragEnd`/`onTransitionEnd`/`onDidPresent` + trailing-таймер (библиотечный `transitionend` на drag-snap ненадёжен — браузерный баг). Проброшен `onDrag` как pass-through проп.

### Изменено

* **`CupertinoPane` — тема и z-index по токенам кита.** Фон/текст/хэндл/destroy-кнопка маппятся на M3-токены (`--color-surface-container`/`-on-surface`/…), шторка следует светлой/тёмной теме приложения вместо хардкод-белого. `.pane`/`.backdrop` подняты в overlay-диапазон кита (z-50/z-40), чтобы шторка была над page-chrome (FAB z-20, контролы карты z-10).

### Версия 0.1.4

Провайдер-независимые контролы карты (kit-rendered M3), escape-hatch для нативных опций, фиксы темы хрома и загрузки кластеризатора.

### Добавлено

* **`MapView` — провайдер-независимые контролы.** Проп `controls: boolean | { attribution, zoom, geolocation, scale, layer }` (общий резолвер `providers/controls.ts`). `attribution`/`scale` — нативные пассивные (Yandex `copyrights`/`YMapScaleControl`, Google `disableDefaultUI`/`scaleControl`); `zoom`/`geolocation`/`layer` — **kit-rendered M3-кнопки** (`maps/controls/`), которые дёргают карту через новый императивный API `MapHandle` (`zoomIn`/`zoomOut`, `getLayer`/`setLayer`/`availableLayers`). Geolocation — на стороне кита через `navigator.geolocation` → `setCenter`, нативный контрол не нужен. `availableLayers()` отдаёт только поддержанные провайдером типы (Yandex map/satellite, Google map/satellite/hybrid) — свитчер не предлагает врущих тогглов. Inline-SVG в кнопках (кит по дизайну не тянет icon-сеты).
* **`MapView` — `controlsPosition`** (`top-left|top-right|bottom-left|bottom-right`, дефолт `bottom-right`) для оверлея контролов.
* **`MapViewOptions.providerOptions`** — escape-hatch для нативных опций провайдера без кросс-провайдерного аналога, по ключу провайдера (`{ yandex: {…}, google: {…} }`); применяется подобъект активного провайдера, мержится в нативную карту последним (перекрывает нормализованные пропы).

### Изменено

* **`MapProviderName`** — единый источник в `providers/types.ts` (раньше дублировался в `providers/index`), ре-экспорт сохранён.
* **Google-провайдер** — симметрия с Yandex: нормализованные `controls`/`scale`/`layer` (`mapType`)/`zoom`.
* **`MapConfig`** — обновляет поля `config` гранулярно, а не заменяет объект целиком: смена темы больше не инвалидирует читателей `apiKey`/`lang`/`mapId`, поэтому `MapView` ретинтит карту на месте, а не пересоздаёт её (пересоздание переподключало кластеризатор и роняло маркеры на каждый тоггл темы).

### Исправлено

* **Тема хрома `YMap`.** Тема задавалась только слою тайлов (`YMapDefaultSchemeLayer`), а копирайт/distribution/контролы остаются на теме самого `YMap` — в dark-mode хром оставался светлым (белый фон кнопки) и наследовал белый текст приложения → «Открыть Яндекс Карты» белым по белому. Тема теперь прокидывается и в `YMap` (и обновляется в `setTheme`).
* **Загрузка кластеризатора Yandex.** `ymaps3.import` без зарегистрированных loader'ов падал с `no loader for pkg @yandex/ymaps3-clusterer` (молчаливый деград на plain-маркеры). Перед импортом регистрируется CDN (`ymaps3.import.registerCdn`) и используется версионированный пакет `@yandex/ymaps3-clusterer@0.0.1`.

`VirtualList` — восстановление позиции скролла по `name`; пресет `ChatList` переименован в `MessagesList`.

### Добавлено

* **`VirtualList` — restore скролла через проп `name`.** По аналогии с `<Content name=…>`: задан `name` → позиция запоминается/восстанавливается между ремаунтами (например, навигация туда-обратно), `name` служит ключом в module-scoped `Map`. Для виртуального списка наивный `scrollTop` неточен (в DOM только видимые строки), поэтому сохраняем virtua `CacheSnapshot` (измеренные высоты) + offset: высоты восстанавливаем через `cache`-проп `VList` (блок `{#key name}` пересоздаёт список с нужным кэшем), offset — через `scrollTo` после маунта с retry до 12 кадров (virtua подключает скроллер на кадр позже и клампит, пока высоты не устаканятся) и подавлением сохранений на время restore. Без `name` поведение прежнее. Экспортирован `clearVirtualScroll(name)`.
